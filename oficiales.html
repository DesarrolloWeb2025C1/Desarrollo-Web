Documentar para trabajar mejor

Comprender la importancia de documentar en un proyecto web, no solo desde lo técnico, sino también en la planificación, comunicación y colaboración.

Practicaremos cómo crear documentos como el Brief, la Minuta de reunión y los Requerimientos funcionales, utilizando herramientas colaborativas como GitHub Wiki o Google Docs.

¿Por qué documentar?

La documentación es una herramienta clave en cualquier proyecto web. No solo sirve para “tener un respaldo”, sino que:

  • Permite registrar decisiones, cambios, acuerdos y tareas.
  • Facilita el trabajo en equipo y la incorporación de nuevos miembros.
  • Evita malentendidos con clientes, tutores o compañeros.
  • Ordena el desarrollo del proyecto y mejora la planificación.

En el contexto de desarrollo web... la documentación actúa como una fuente única de la verdad (Single Source of Truth).

Tipos de documentación en un proyecto web

1. Documentación estratégica y de comunicación

  • Minuta: Registro breve de una reunión que resume lo conversado, los acuerdos y las tareas asignadas.

Objetivos de una minuta:

  • Tomar notas de una reunión.
  • Dejar asentada la información revisada.
  • Registrar los acuerdos alcanzados.
  • Definir acciones a tomar.
  • Indicar los participantes.
  • Dar seguimiento a acciones previas.

Cómo estructurar una minuta:

  1. Asignar un responsable antes de la reunión.
  2. Preparar una plantilla con los temas a tratar.
  3. Escuchar y tomar nota.
  4. Revisar las notas.
  5. Elaborar conclusiones y enviarlas.
  • Brief: Documento inicial que sintetiza el objetivo del proyecto, su público y el problema a resolver.

Elementos de un Brief:

  • Descripción del proyecto.
  • Datos de contacto.
  • Público objetivo.
  • Productos o servicios.
  • Competencia.
  • Referencias de diseño.
  • Objetivos primarios y secundarios.
  • Presupuesto estimado.

2. Documentación de requerimientos

Especifica lo que el sitio debe hacer y cómo debe comportarse.

  • Funcionales: describen funciones y comportamientos esperados.
  • No funcionales: definen características de calidad y restricciones.

Ejemplos funcionales:

  • Mapa del sitio.
  • Registro y autenticación de usuarios.
  • Carrito de compras y checkout.
  • Integración con pasarelas de pago.

Ejemplos no funcionales:

  • Tiempo de carga menor a 2 segundos.
  • Compatibilidad con navegadores principales.
  • Cumplimiento de estándares WCAG 2.1 AA.

3. Documentación técnica

Describe cómo se construirá la web y qué se necesita para hacerlo.

  • Mapa del sitio detallado con rutas y plantillas.
  • Especificaciones técnicas (HTML, CSS, JS, CMS, frameworks).
  • Compatibilidad y navegadores soportados.
  • Guía de estilo visual.
  • Normas de desarrollo y despliegue.

Herramientas recomendadas

  • GitHub Wiki: documentación con control de versiones.
  • Google Docs: edición colaborativa.
  • HackMD: edición de Markdown en equipo.
  • Notion: paneles visuales y organización.

Arquitectura de una aplicación web - Diseñando un sitio

La arquitectura de una aplicación web es un tema importante cuando se trata de la participación del usuario. La arquitectura de aplicaciones web permite a los desarrolladores diseñar aplicaciones web que satisfagan las necesidades básicas y avanzadas de los visitantes. Los criterios pueden variar desde una carga rápida hasta una navegación simple.

Un objetivo importante para cualquier negocio es evitar pagar mucho dinero para desarrollar una aplicación. Para hacerlo, primero debe comprender qué constituye una buena arquitectura.

Para ello debería plantearme varias preguntas al momento de planificar el desarrollo:

  • ¿El sitio se ejecuta solo en el browser?
  • ¿Va a hacer uso de recursos subidos en un servidor?
  • ¿Es su arquitectura apta para el futuro?
  • ¿Es escalable y sostenible?
  • ¿Y cómo decidir qué componentes y funcionalidades incluir en una aplicación?

¿La aplicación se ejecuta en el browser?

No siempre, muchas se ejecutan en un servidor web, es decir usan recursos subidos a un servidor de aplicaciones y las aplicaciones acceden a través de una conexión web. La aplicación se crea con lenguajes de programación como Javascript, CSS y HTML mientras que los recursos subidos en un servidor de aplicaciones se crean con lenguajes de programación como Java, Visual C#, PHP, Python, etc.

Las tendencias, las modas o las tecnologías del mercado cambian con el tiempo, las expectativas de los clientes también y el crecimiento de una empresa nunca termina.

Es importante definir una arquitectura que permita que la aplicación sea escalable, sólida y sostenible.

Arquitectura de una aplicación “Modelo de capas”

La arquitectura o modelo de capas se trata de un modelo de interacciones simultáneas entre componentes, bases de datos, sistemas de middleware (que conectan diferentes servicios), interfaces de usuario y servidores en una aplicación.

La arquitectura en capas consta en dividir la aplicación en capas, con la intención de que cada capa tenga un rol muy definido, como podría ser, una capa de presentación (capa de interfaz de usuario), una capa de reglas de negocio (capa de negocio) y una capa de acceso a datos (capa de datos). Este estilo arquitectónico no define cuántas capas debe de tener la aplicación, sino más bien se centra en la separación de la aplicación en capas. Algunos incluyen una cuarta capa, denominada capa de entidades donde se encuentran las representaciones de datos que atraviesan todas las capas (ejemplo: un alumno, un empleado, aulas, ventas).

En una arquitectura en capas, todas las capas se colocan de forma horizontal, de tal forma que cada capa solo puede comunicarse con la capa que está inmediatamente por debajo. Por ejemplo, si la capa de presentación requiere consultar la base de datos, tendrá que solicitar la información a la capa de negocio, la cual la solicitará a la capa de persistencia, y esta a su vez a la base de datos. Finalmente, la respuesta retornará en sentido inverso hasta llegar a la capa de presentación.

Descripción de cada capa

Capa de interfaz de usuario

La interfaz de usuario es el cómo mostrar los datos de la aplicación en la pantalla y tiene como principal función ser el punto principal de interacción con el usuario. Cuando los datos cambian (ya sea por interacción del usuario o por entrada externa), deben actualizarse para reflejar los cambios.

Aunque los datos reales se encuentren almacenados en una base de datos, la interfaz es una representación visual del estado de esos datos. Un tema a considerar es que a esta capa no pueden llegar datos sensibles que no sean necesarios para la funcionalidad mostrada.

Capa de negocio

Es la responsable de encapsular la lógica de dominio compleja o la lógica de negocio, la toma de decisiones, los algoritmos, el entrecruzamiento de datos. Esta capa transforma a los datos en la información que consumirá la capa de interacción con el usuario.

Puede ser reutilizada por varios dispositivos en simultáneo: móviles, tablets o computadoras. Aquí se establecen todas las reglas que deben cumplirse.

Capa de base de datos

La capa de datos está formada por repositorios que administran las fuentes de datos. Cada repositorio sabe conectarse a una base de datos, con el objetivo de leer, ingresar, modificar y eliminar datos.

Ventajas y desventajas de la arquitectura de capas

Ventajas

  • Separación de responsabilidades: cada capa tiene una sola responsabilidad.
  • Fácil de desarrollar: muy conocida y usada.
  • Fácil de probar: se pueden probar capas de forma individual.
  • Fácil de mantener: permite detectar errores de forma más clara.
  • Seguridad: aislamiento de capas dificulta ataques.

Desventajas

  • Performance: la comunicación entre capas puede degradar el rendimiento.
  • Complejidad de despliegue: los componentes dependen de ser desplegados en orden.
  • Tolerancia a fallos: si una capa falla, las superiores también fallan en cascada.

Front-End vs Back-End

En el desarrollo se habla mucho de desarrolladores front-end y back-end, según la capa en la que se especialicen.

Front-End: se enfoca en la interfaz de usuario. Responde a la entrada del usuario y genera la representación visual (HTML, CSS, JavaScript en apps web).

Back-End: se enfoca en la capa de negocio y de datos. Sus funciones son:

  • 1- responder a solicitudes HTTP y devolver respuestas,
  • 2- obtener, modificar, almacenar o eliminar datos.
    • Se puede programar en C#, Java, Python, Ruby, Node.js, PHP, entre otros.

¿Qué es un repositorio?

Un repositorio es un almacenamiento virtual de tu proyecto. Te permite guardar versiones del código a las que puedes acceder cuando lo necesites.

Este sistema nos permite, mantener nuestro código guardado en un repositorio local o en un repositorio remoto, un repositorio local se encuentra en nuestra computadora y un repositorio remoto en la nube.



El repositorio remoto es centralizado, es decir todos podemos acceder a él mientras tengamos los permisos necesarios. El repositorio local aunque es una copia, de esta solo guardado en la computadora del usuario.

Cuando tenemos un repositorio remoto, podemos obtenerlo con un comando llamado clonar.



Control de versiones 

Una vez un alumno de Comercio Electrónico de una universidad en José C. Paz desarrolló un sitio web en HTML, CSS y JS durante varias horas. Se trataba de un proyecto final “un e-commerce”, como parte del trabajo práctico de un curso de desarrollo. Resulta que una noche antes de la entrega del proyecto, él tenía funcionando el sitio con un buen avance de los requisitos requeridos; sin embargo, decidió hacer unos arreglos finales a últimas horas de la noche. El profesor les había advertido que los cambios de último momento generan fallas inesperadas.

Pero el alumno hizo caso omiso a la recomendación, pensando que a un buen desarrollador  no le pasaban esas cosas!... La historia concluye en que a la hora de entregar el proyecto, el programa falló y el alumno se lamentó diciendo: si no hubiera hecho esos últimos cambios, mi programa no hubiera fallado!!.

El alumno debió tener un sistema de control de versiones listo para hacer este trabajo práctico.

Existen muchos Sistemas de Control de Versiones siendo algunos de los más conocidos CVS, Git, Subversion, Mercurial y Bazar. 

Elegimos Git, porque es hoy en día uno de los sistemas más populares.

Git es un Sistema de control de versiones distribuido, que realiza un seguimiento de cada modificación realizada en el código a lo largo del tiempo, y le permite  retroceder si es necesario y deshacer esos cambios. Git le permitirá volver a un estado anterior en un proyecto o ver toda su evolución desde que se creó el proyecto. Git también se llama gestión de código fuente.

Esto significa que cuando se realiza un cambio en un archivo, Git guardará la versión anterior y registrará exactamente qué modificaciones se hicieron.

Este proceso se repite cada vez que se modifica el archivo, lo que te permite volver a cada punto en el historial de un proyecto y recuperar fácilmente una versión específica. En pocas palabras, esto te brinda un número ilimitado de acciones de “deshacer”, para que nunca pierdas nada. Además, te permite a ti (y a otros) ver cómo se ha desarrollado un proyecto con el tiempo.

¿Qué son las API?

Las API son mecanismos que permiten a dos componentes de software comunicarse entre sí mediante un conjunto de definiciones y protocolos.

Por ejemplo, el sistema de software del instituto de meteorología contiene datos meteorológicos diarios. La aplicación meteorológica de su teléfono “habla” con este sistema a través de las API y le muestra las actualizaciones meteorológicas diarias en su teléfono.

¿Qué significa?

API significa “interfaz de programación de aplicaciones”. En este contexto, la palabra aplicación se refiere a cualquier software con una función distinta. La interfaz puede considerarse como un contrato de servicio entre dos aplicaciones. Este contrato define cómo se comunican entre sí mediante solicitudes y respuestas.

La documentación de una API contiene información sobre cómo los desarrolladores deben estructurar esas solicitudes y respuestas.

¿Cómo funcionan las API?

La arquitectura de las API suele explicarse en términos de cliente y servidor. La aplicación que envía la solicitud se llama cliente, y la que envía la respuesta se llama servidor. En el ejemplo del tiempo, la base de datos meteorológica del instituto es el servidor y la aplicación móvil es el cliente.

En general, el funcionamiento de una API es:

  1. La API se aloja en un servidor y recibe solicitudes de otros sistemas.
  2. El cliente envía una solicitud mediante un protocolo como HTTP.
  3. La API procesa la solicitud y devuelve una respuesta (datos o acción).
  4. La respuesta suele estar en formato estructurado como JSON o XML.
  5. Puede incluir autenticación y autorización para controlar el acceso.

Protocolo de comunicación

  • HTTP: el más común, sencillo de implementar y compatible con cualquier lenguaje.
  • REST: basado en HTTP, utiliza verbos como GET, POST, PUT y DELETE.
  • SOAP: más complejo, usa XML. Común en entornos empresariales.
  • GraphQL: permite definir qué datos recuperar, más flexible.

El contrato de la API

Es un conjunto de reglas, especificaciones y documentación que describe cómo los clientes pueden interactuar con la API. Incluye sintaxis de solicitudes, tipos de datos, códigos de estado, parámetros, limitaciones, autenticación, etc.

¿Qué son las API docs?

Son la documentación técnica de una API. Describen contrato, métodos, parámetros, códigos de respuesta y ejemplos de uso. Son clave para que los desarrolladores utilicen la API correctamente.

Clasificación de las API

Según su acceso

  • APIs Privadas: uso interno dentro de una organización.
  • APIs Públicas: disponibles para cualquier desarrollador.

Según su aplicación

  • APIs de datos: exponen conjuntos de datos (CRUD).
  • APIs de sistemas operativos: permiten interactuar con funciones del SO.
  • APIs Web: comunicación entre aplicaciones web vía HTTP.

Ejemplos de API

  • PayPal API: pagos en línea.
  • Google Maps API: integración de mapas.
  • OpenWeatherMap API: datos meteorológicos.
  • WhatsApp API: envío de mensajes.

Visual Studio Code


Visual Studio Code (VS Code) es un editor de código fuente gratuito y multiplataforma desarrollado por Microsoft, conocido por su ligereza y potencia. Ofrece soporte para múltiples lenguajes de programación, integración con sistemas de control de versiones como Git, y una amplia variedad de extensiones que permiten personalizar y extender sus funcionalidades.

Extensiones para desarrollo

  • Vscode-icons: Esta extensión añade un conjunto de íconos atractivos para archivos y carpetas en el explorador de Visual Studio Code. Ayuda a identificar rápidamente los tipos de archivos y mejorar la organización visual dentro del editor.

  • Mithril Emmet: Esta extensión ofrece soporte de Emmet para el framework Mithril.js, permitiendo a los desarrolladores escribir HTML de manera más rápida y eficiente con abreviaturas que se expanden en estructuras HTML completas. Es especialmente útil para quienes trabajan con este framework específico.

  • Image preview: Esta extensión permite previsualizar imágenes directamente en Visual Studio Code. Al colocar el cursor sobre una ruta de imagen en el código, aparecerá una vista previa de la imagen, lo que es útil para proyectos que manejan muchos recursos gráficos.

  • Github Copilot Chat: Usa inteligencia artificial para ayudarte a programar.
    Podés pedirle explicaciones, ejemplos o generar código escribiendo directamente en un chat dentro de VS Code.


Extensiones importantes

  • Live Server: Live Server es una popular extensión que lanza un servidor de desarrollo local con recarga en vivo para páginas web estáticas o dinámicas. Esto significa que cada vez que guardas un archivo, la página se recarga automáticamente en el navegador, lo que acelera el flujo de trabajo al ver los cambios en tiempo real.

  • CSS Peek: Permite ver y acceder al código CSS relacionado con una clase o ID en tus archivos HTML.
    Podés mantener el cursor sobre el nombre de la clase y ver el estilo sin tener que abrir manualmente el archivo CSS (Ctrl + cursor).
  • Error Lens: Resalta directamente en el código los errores y advertencias (como variables sin usar o errores de sintaxis).
    Así no tenés que mirar la consola inferior: el problema aparece en línea, justo donde ocurre.
  • Spanish Language Pack: Traduce la interfaz de Visual Studio Code al español (menús, mensajes, botones, etc.).
    Ideal para quienes recién comienzan y prefieren usar el entorno en su idioma.

Formulario funcional con API - Formpress

Formspree es un servicio en línea o API que nos permite dar funcionalidad a formularios de páginas web estáticas, sí, páginas web con solo HTML y CSS.

Para integrar Formspree a nuestro sitio web no es necesario de conocer o dominar ningún lenguaje de programación ni programar absolutamente del lado del backend. Así que nos olvidamos de PHP, Python, Ruby, JavaScript etc..




https://formspree.io/

¿Cómo generar un formulario funcional con Formpress?

Primero deberán crear una cuenta en el en el sitio web de formspree. Es completamente gratis. Una vez que hayan verificado la cuenta ya serán capaces de hacer uso de formspree.

Por lo cual, lo siguiente será ir a Crear un nuevo proyecto:


 Una vez generado el proyecto, deberán ir a Agregar nuevo y crear el formulario para conseguir el código para integrar la API en tu sitio:




Una vez creado el Formulario se les generará un código o link para conectar el formulario que  hiciste en tu HTML con la API.




Insertar la API en el HTML

Para que Formspree maneje tu formulario, necesitarás agregar un atributo action a tu elemento <form> junto con el link que te proporcionó el sitio y además agregar un method="POST"  

<form action="//formspree.io/youcodeform" method="POST">

Lo siguiente es dar a cada campo un atributo name a los inputs generados; éstos serán las etiquetas que aparecen cada vez que recibes un envío del formulario.



 Ya con eso, lograrás un formulario funcional para que recibas correos para quien complete tu formulario de contacto!
